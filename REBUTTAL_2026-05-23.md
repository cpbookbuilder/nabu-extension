## Critical

### 1) `/restore` entitlement takeover — **FIXED**

Replaced single-step email-only restore with a two-step Stripe-portal-verified flow.

- **POST `/api/extension/restore` `{email, device_id}`** now does NOT mutate any local state. It looks up a Stripe customer by email and, if a billable subscription is present, returns a Stripe Customer Portal URL whose `return_url` carries a short-lived JWT ticket binding `(device_id, customer_id)`.
- **GET `/api/extension/restore-complete?ticket=...`** is the only path that mutates entitlement. Stripe's portal authenticates the user via an email-link OTP before redirecting back here, so the holder of the ticket has proven ownership of the email Stripe has on file. We re-verify the subscription is still billable, detach Pro from any other local row mapped to the same Stripe customer, and attach Pro to `device_id`.
- Ticket TTL: 15 minutes; signed with the same `JWT_SECRET` used for device tokens; `type: "restore"` claim prevents reuse as a device token.

**Code:** `backend/extension_routes.py` — `POST /restore`, `GET /restore-complete`, `_BILLABLE_STATES`, `RESTORE_TICKET_TTL_MIN`.
**Frontend:** `chrome-extension/popup.js` and `chrome-extension/dashboard.js` — restore now opens `data.verify_url` in a new tab and instructs the user to refresh.
**Tests:** `backend/tests/test_restore_and_billing.py` — covers no-match, no-billable-subs, happy path, 503-when-no-Stripe, invalid ticket, expired ticket, transfer-to-target-device, and detach-from-other-device.

Email-existence enumeration via `/restore` is still *partially* observable because the response contains `verify_url` only when a match exists. This is bounded by the 5-per-IP/hour rate limit. We documented the trade-off; mitigating fully would require returning a decoy portal URL (Stripe doesn't let us forge one) or an out-of-band email channel (no infra today). Acceptable risk given the takeover vector is closed.

### 2) SPA route-change persistence race — **FIXED**

Snapshot-and-flush before the page key shifts.

- `pageKey(url?)`, `saveThreads(url?)`, `updateHistoryIndex(records, url?)` now all accept an explicit URL so the caller can target a specific bucket instead of relying on `location.href` at write time.
- `flushSave(url)` cancels the debounce and writes synchronously against `url`.
- `_onLocationChange()` captures `oldUrl = _lastUrl`, calls `flushSave(oldUrl)` *before* tearing down threads and updating `_lastUrl`, then closes threads with `{skipSave: true}` so no second save fires against the new URL.
- `closeThread(id, {skipSave})` honors the skip flag — only the route-change path passes it.

**Code:** `chrome-extension/content.js` — `pageKey`, `flushSave`, `saveThreads`, `updateHistoryIndex`, `closeThread`, `_onLocationChange`.

## High

### 3) Retention by `created_at` vs activity — **FIXED**

Added `ExtensionUser.last_seen_at`, bumped on every register / `/usage` / `/annotate`, and switched the class-2 inactivity purge predicate to use it.

- New nullable `last_seen_at` column with a Postgres-only idempotent backfill (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, then `UPDATE ... SET last_seen_at = created_at WHERE last_seen_at IS NULL`) so legacy rows survive the first purge after the migration.
- Purge predicate: `COALESCE(last_seen_at, created_at) < cutoff` — defensive in case the backfill hasn't run yet.
- Touch points: `register` (both new-user and refresh-token branches), `get_usage`, `annotate` (both increment and create-row branches).

**Code:** `backend/db_models.py`, `backend/db.py::create_tables`, `backend/extension_routes.py::register`, `get_usage`, `annotate`, `purge_old_data`.
**Tests:** `test_register_sets_last_seen_at`, `test_annotate_bumps_last_seen_at`.

Also added the Postgres-dialect guard in `create_tables` (`if conn.dialect.name != "postgresql": return`) so the test SQLite engine doesn't choke on `DO $$ ... END $$;`.

### 4) Stripe cancellation only covers `active` — **FIXED**

`delete_account` now lists `status="all"` and cancels any subscription whose status is in `{trialing, active, past_due, unpaid, incomplete, paused}`. Terminal states (`canceled`, `incomplete_expired`) are skipped — no point cancelling something already terminal.

**Code:** `backend/extension_routes.py::delete_account` (added `CANCELLABLE` set).
**Test:** `test_delete_account_cancels_all_billable_states_not_just_active` — sets up four subs (`active`, `trialing`, `past_due`, `canceled`), asserts exactly 3 are deleted.

### 5) `restored` boolean leaks entitlement existence — **FIXED (collapsed into #1)**

The new `/restore` response shape drops the `restored` field entirely. Returns either `{message}` (no match) or `{message, verify_url}` (match — must complete via Stripe). Both popup.js and dashboard.js branch on `verify_url` presence, not on `restored`. Email-existence inference via `verify_url` is documented under #1 above.

### 6) Popup/dashboard session staleness after JWT expiry — **REBUT**

Defensible as-is for now. The current behavior:

- `popup.js::loadUsage` reads `annotate_jwt`; if missing/invalid, shows "Not connected yet — open a page and ask a question." That is *correct* UX guidance, not a broken state.
- The content script re-establishes the session on first annotate via `getSession()` (clears stale JWT on 401, registers fresh). So the session always heals on next interaction.

Implementing a shared `ensureSession()` for popup/dashboard would:
- Add an extra unattended network call every time the popup opens (most opens are read-only).
- Push first-use friction into the popup, which currently shows instantly.
- Add a code path that has to deal with `register` failures *in the popup*, where there is no thread context to recover into.

If the user's reported pain ever becomes "I have to ask a question to fix the popup," this is a one-day fix. No reports yet; deferring.

Tracking: `TODO.md` entry to be added when the first user reports stale state.

### 7) Debounced persistence without unload flush — **FIXED**

Added `pagehide` listener that calls `flushSave(location.href)` synchronously. Chose `pagehide` over `beforeunload` because:

- `beforeunload` doesn't fire on Safari/iOS or on bfcache evictions.
- `pagehide` fires reliably on all modern browsers and is the documented signal for "page is about to leave the live state."

Also covers the route-change case as a side effect of the #2 fix.

**Code:** `chrome-extension/content.js` — `window.addEventListener('pagehide', () => flushSave(location.href))`.

### 11) Hash-router URL keying — **REBUT (documented for follow-up)**

`pageKey()` strips the hash. For the *overwhelming majority* of pages, the hash is an in-page anchor (`#section-2`) and threads anchored to the same article should restore regardless of which section is in view. Treating hash as part of the key would orphan threads on every `#section` change.

The minority of true hash-routed SPAs (some old Angular apps, some doc viewers) would benefit from per-route keying. But:
- Heuristically detecting "hash is a route" vs "hash is an anchor" is hard (no universal signal).
- A static allowlist of known hash-routed domains is brittle and would need maintenance.
- The current behavior degrades gracefully: threads attach to the page broadly, not the specific route.

Not blocking CWS submission. Tracked as `TODO: hash-router site-specific keying` — revisit if a real user complains about thread bleed across hash routes on a specific site.

### 12) Runtime KaTeX CSS fetch — **REBUT (documented for follow-up)**

Yes, KaTeX CSS + fonts are fetched from jsDelivr at runtime. The trade-off:

- The CDN URL is pinned to `@0.16.11` (immutable per jsDelivr's policy).
- Bundling fonts adds ~200KB to the extension package (KaTeX ships ~14 font files for various weights/scripts).
- Sites with strict CSPs would block the CDN fetch and silently fall back to unstyled equation HTML — bundling would fix this, but those are also sites where the floating button/popover often fail anyway.

Acceptable now. Tracked as `TODO: bundle KaTeX fonts` — revisit if a CWS reviewer flags the runtime fetch, or if a user reports broken equations on a CSP-strict site.
